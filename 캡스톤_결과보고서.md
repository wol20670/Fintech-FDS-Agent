## 1. 프로젝트 개요

### 1-1. 프로젝트명

**실시간 지능형 사기 탐지 및 자율 대응 에이전트 (Fintech FDS Agent v1)**

### 1-2. 기획 배경

최근 몇 년 사이 비대면 금융 거래가 폭발적으로 증가하였음. 한국은행 통계에 따르면 모바일뱅킹과 인터넷뱅킹을 통한 거래 건수는 매년 두 자릿수 성장률을 기록하고 있으며, 이에 비례하여 금융 사기 수법도 빠르게 지능화되고 있습니다.

기존에 금융권에서 사용하던 **규칙 기반(Rule-Based) FDS**는 "1일 이체 한도 초과 시 차단", "새벽 시간대 고액 출금 시 알림" 같은 단순 조건문 방식이었음. 이 방식은 운영자가 직접 규칙을 손으로 짜야 하고, 사기꾼이 규칙을 학습해 우회하면 바로 무력화된다는 치명적인 한계가 있습니다.

이 프로젝트를 기획하게 된 가장 큰 이유도 바로 이 지점이었음. 수업 시간에 배운 머신러닝 모델이 단순 규칙보다 훨씬 복잡한 패턴을 학습할 수 있다는 점에 착안하여, **ML 기반의 자율 판단 시스템**을 직접 구현해보고자 하였습니다.

### 1-3. 핵심 목표

본 프로젝트의 핵심 목표는 세 가지로 정리할 수 있음.

첫째, 실시간으로 들어오는 거래 데이터를 분석하여 사기 여부를 **100ms 이내**에 판단하는 고속 추론 시스템을 구축하는 것입니다.

둘째, 단순히 "사기다/아니다"를 알려주는 것을 넘어, 위험 점수(Risk Score)에 따라 **자동 승인 → 추가 인증 요구 → 거래 차단 → 계좌 동결**의 4단계 자율 대응을 수행하는 에이전트를 만드는 것이었음.

셋째, 모델이 왜 그런 판단을 내렸는지 사람이 이해할 수 있도록 **설명 가능한 AI(XAI)** 의 기초를 구현하는 것입니다. 금융 시스템에서 AI의 판단을 무조건 신뢰하기보다는, 근거를 함께 제공하여 **신뢰성 확보**를 하는 것이 중요하다고 판단하였음.

### 1-4. 주요 적용 대상

은행, 카드사, 핀테크 기업 등 금융 서비스를 제공하는 기관이 주요 대상임. 특히 모바일 뱅킹, 간편 결제 플랫폼, 이커머스 보안 영역에서 활용 가능성이 높습니다.

## 2. 데이터 활용 및 처리

### 2-1. PaySim 데이터셋을 선택한 이유

실제 금융 거래 데이터는 개인정보 보호 규정(개인정보보호법, GDPR 등)으로 인해 외부에서 구하는 것이 사실상 불가능함. 따라서 우리 팀은 Kaggle에 공개된 **PaySim**이라는 합성(Synthetic) 금융 거래 데이터셋을 활용하였습니다.

PaySim은 실제 아프리카 모바일 금융 서비스의 거래 패턴을 통계적으로 모방하여 생성된 데이터셋으로, 총 **630만 건 이상**의 거래 기록을 포함하고 있음. 실제 데이터의 분포와 패턴을 최대한 재현하고 있어, 학부 수준의 프로젝트에서 현업에 가까운 환경을 시뮬레이션하기에 적합하다고 판단하였습니다.

데이터셋의 주요 컬럼 구성은 다음과 같음.

| 컬럼명 | 설명 |
|--------|------|
| `step` | 시뮬레이션 시간 단위 (1 = 1시간) |
| `type` | 거래 유형 (CASH_IN / CASH_OUT / DEBIT / PAYMENT / TRANSFER) |
| `amount` | 거래 금액 |
| `nameOrig` | 송금인 계좌 ID |
| `oldbalanceOrg` | 송금인 거래 전 잔액 |
| `newbalanceOrig` | 송금인 거래 후 잔액 |
| `nameDest` | 수취인 계좌 ID |
| `oldbalanceDest` | 수취인 거래 전 잔액 |
| `newbalanceDest` | 수취인 거래 후 잔액 |
| `isFraud` | 실제 사기 여부 (0: 정상, 1: 사기) |

### 2-2. 데이터 불균형 문제와 해결 방법

이 프로젝트에서 가장 먼저 마주친 큰 벽이 바로 **데이터 불균형(Class Imbalance)** 문제였음. 전체 630만 건의 거래 중 실제 사기 거래는 고작 약 0.13%, 즉 8,000건 남짓에 불과했습니다.

이 상태로 그냥 모델을 학습시키면 어떤 일이 벌어지는지 처음에는 잘 몰랐음. 직접 해보니, 모델이 그냥 모든 거래를 "정상"이라고 예측해버리는 문제가 발생하였습니다. 정확도(Accuracy)는 99.87%로 높게 나오지만, 사기를 하나도 못 잡는 쓸모없는 모델이 되는 것이었음. 수업에서 배운 "정확도의 역설(Accuracy Paradox)"을 직접 체험한 순간이었습니다.

이를 해결하기 위해 **하이브리드 샘플링 전략**을 적용하였음. 단계는 아래와 같습니다.

**1단계 — RandomUnderSampler (언더샘플링)**

정상 거래 데이터를 사기 거래의 10배 수준으로 줄임. 전체 데이터 크기를 먼저 줄여서 이후 SMOTE 연산 비용을 감당 가능한 수준으로 낮추는 전략이었습니다.

```python
# 정상:사기 비율을 10:1로 조정
under = RandomUnderSampler(sampling_strategy=0.1, random_state=42)
```

**2단계 — SMOTE (오버샘플링)**

SMOTE(Synthetic Minority Over-sampling Technique)는 단순히 사기 데이터를 복사하는 것이 아니라, 기존 사기 샘플들 사이에 새로운 가상의 샘플을 **보간(Interpolation)** 하여 생성하는 기법임. 덕분에 단순 복사보다 모델이 더 다양한 사기 패턴을 학습할 수 있습니다.

```python
# 언더샘플링 후, 사기 데이터를 정상 데이터의 50% 수준까지 증식
over = SMOTE(sampling_strategy=0.5, random_state=42)

# Pipeline으로 두 단계를 연결
pipeline = Pipeline(steps=[('u', under), ('o', over)])
X_res, y_res = pipeline.fit_resample(X_train, y_train)
```

이렇게 하이브리드 전략을 쓴 이유는, SMOTE만 단독으로 쓸 경우 원본 데이터가 너무 커서 메모리 초과(OOM) 오류가 발생했기 때문이었음. 언더샘플링으로 전체 규모를 줄인 다음 SMOTE를 적용하는 순서가 핵심이었습니다.

### 2-3. 피처 엔지니어링

원본 데이터에 있는 컬럼만으로는 사기 패턴을 충분히 잡아내기 어려웠음. 금융 도메인 지식을 바탕으로 아래 두 가지 파생 피처를 직접 만들었습니다.

**잔액 불일치 피처 (Balance Discrepancy)**

정상적인 거래라면 "거래 전 잔액 - 거래 금액 = 거래 후 잔액"이 정확히 성립해야 함. 그런데 PaySim 데이터를 분석해보니, 사기 거래의 상당수에서 이 등식이 성립하지 않는 경우가 많았습니다. 이 불일치를 숫자로 표현한 피처가 `balance_diff_orig`와 `balance_diff_dest`임.

```python
# 송금인 쪽 잔액 불일치: 0에 가까울수록 정상, 멀수록 의심
df['balance_diff_orig'] = df['oldbalanceOrg'] - df['newbalanceOrig'] - df['amount']

# 수취인 쪽 잔액 불일치
df['balance_diff_dest'] = df['newbalanceDest'] - df['oldbalanceDest'] - df['amount']
```

**계좌 유형 피처 (Account Type)**

PaySim에서 계좌 ID는 'C'(개인고객)나 'M'(가맹점)으로 시작함. 사기는 주로 개인 계좌 사이에서 발생하고, 가맹점 계좌가 수취인인 경우에는 사기 확률이 낮다는 패턴을 발견하였습니다.

```python
# 개인 고객 계좌 여부 (1: 개인, 0: 가맹점)
df['orig_is_customer'] = (df['nameOrig'].str[0] == 'C').astype('int8')
df['dest_is_merchant']  = (df['nameDest'].str[0] == 'M').astype('int8')
```

이 외에도 FastAPI 백엔드의 FDS Metadata 영역에는 **최근 1시간/24시간 내 거래 횟수(`tx_count_1h`, `tx_count_24h`)**, **접속 위치(`geo_location`)**, **신규 디바이스 여부(`is_new_device`)** 등의 파생 피처를 추가로 정의하여, 실제 운영 환경에서 더욱 풍부한 판단 근거를 확보할 수 있도록 설계하였음.

또한 데이터 로딩 시에는 630만 건이라는 대용량을 처리하기 위해 `chunksize=200,000`으로 분할 로딩하고, 컬럼별 데이터 타입을 `int16`, `float32` 등으로 최적화하여 메모리를 약 40% 절감하였습니다.

### 3-2. 모델 학습 구조 — Stacking Ensemble

단일 모델보다 여러 모델을 조합하는 **스태킹 앙상블** 방식을 채택하였음. 이유는 XGBoost와 LightGBM이 각각 다른 방식으로 데이터를 학습하기 때문에, 둘의 예측을 합치면 오류가 서로 상쇄되어 더 안정적인 결과를 낼 수 있기 때문입니다.

```
[입력 데이터]
     │
     ├─── XGBoost ────────→ 사기 확률 P1
     │
     └─── LightGBM ───────→ 사기 확률 P2
                                  │
                         [P1, P2] 결합
                                  │
                    Logistic Regression (메타 모델)
                                  │
                          최종 사기 확률 (0.0 ~ 1.0)
```

```python
# Base Learner 1: XGBoost
xgb_m = XGBClassifier(n_estimators=100, max_depth=6,
                       tree_method='hist', random_state=42)
xgb_m.fit(X_res, y_res)

# Base Learner 2: LightGBM
lgbm_m = LGBMClassifier(n_estimators=100, random_state=42, verbose=-1)
lgbm_m.fit(X_res, y_res)

# 각 모델의 예측 확률을 메타 모델의 입력으로 사용
p1 = xgb_m.predict_proba(X_test)[:, 1]
p2 = lgbm_m.predict_proba(X_test)[:, 1]
meta_X = np.column_stack([p1, p2])

# Meta Learner: Logistic Regression
meta_model = LogisticRegression().fit(meta_X, y_test)
```

### 3-3. 모델 평가 결과

단순 정확도(Accuracy)보다 **Recall(사기 탐지율)** 을 핵심 지표로 설정하였음. 금융 보안에서는 사기를 놓치는 것(False Negative)이 정상을 사기로 잘못 판단하는 것(False Positive)보다 훨씬 큰 피해를 야기하기 때문입니다.

| 평가 지표 | 결과 | 의미 |
|-----------|------|------|
| ROC-AUC | **0.9996** | 정상/사기 구분 능력이 거의 완벽한 수준 |
| Recall (사기 탐지율) | **99.76%** | 실제 사기 1,000건 중 997건을 잡아냄 |
| Precision (탐지 정확도) | **74.57%** | 사기로 판단한 것 중 약 75%가 실제 사기 |
| F1-Score | **0.8534** | Precision과 Recall의 조화 평균 |
| Accuracy | **99.96%** | 전체 거래 중 정답 비율 |

Precision이 100%가 아닌 것, 즉 오탐(정상을 사기로 판단)이 약 25% 발생하는 것은 처음에는 아쉬운 결과로 보였음. 그러나 금융 보안 관점에서는 오탐이 발생해도 추가 인증 절차를 요청하는 정도의 불편함에 그치지만, 미탐(사기를 놓침)이 발생하면 실제 금전 피해로 이어지기 때문에 **높은 Recall을 우선시하는 것이 올바른 판단**이라고 결론 지었습니다.

모델 평가에는 Matplotlib을 활용하여 PR Curve(Precision-Recall Curve)와 Confusion Matrix를 시각화하였음. ROC-AUC보다 PR-AUC가 불균형 데이터에서 더 객관적인 지표라는 것도 이번 프로젝트를 통해 배웠습니다.

```python
# PR Curve 시각화
precision, recall, _ = precision_recall_curve(y_test, y_probs)
pr_auc = auc(recall, precision)

plt.plot(recall, precision, label=f'PR-AUC = {pr_auc:.4f}')
plt.title('Precision-Recall Curve')
plt.xlabel('Recall')
plt.ylabel('Precision')
```

## 4. 자율 대응 시나리오

### 4-1. 에이전트의 의사결정 구조

FDS 에이전트는 단순히 사기 여부만 알려주는 것이 아니라, 위험 점수에 따라 **4단계 자율 대응**을 수행하도록 설계하였음. 코드에서는 `model_service.py`의 `_score_to_risk_level()`과 `_determine_action()` 함수가 이 역할을 담당합니다.

```python
# 리스크 점수 → 위험 등급 매핑
def _score_to_risk_level(score: float) -> RiskLevel:
    if score < 0.1:   return RiskLevel.SAFE      # 자동 승인
    elif score < 0.3: return RiskLevel.LOW       # 승인 + 로그 강화
    elif score < 0.6: return RiskLevel.MEDIUM    # 추가 인증(SMS/ARS)
    elif score < 0.85:return RiskLevel.HIGH      # 즉시 차단
    else:             return RiskLevel.CRITICAL  # 계좌 동결

# 위험 등급 → 자율 대응 조치
def _determine_action(risk_level: RiskLevel) -> str:
    action_map = {
        RiskLevel.SAFE:     "APPROVE",
        RiskLevel.LOW:      "APPROVE",
        RiskLevel.MEDIUM:   "HOLD",    # 추가 인증 요청
        RiskLevel.HIGH:     "BLOCK",   # 거래 차단
        RiskLevel.CRITICAL: "FREEZE",  # 계좌 동결
    }
    return action_map[risk_level]
```

### 4-2. 시나리오 1 — 새벽 3시, 해외 IP, 500만원 결제

기획서 초안에서 제시된 가장 대표적인 시나리오임. 실제로 우리 백엔드에서 이 요청이 들어왔을 때 어떻게 처리되는지 단계별로 설명합니다.

### 4-3. 시나리오 2 — 정상적인 소액 결제

반대로 정상 거래는 어떻게 처리되는지도 확인하였음. 평소 자주 이용하는 가맹점에 소액을 결제하는 경우, 에이전트는 수십 밀리초 내에 `SAFE / APPROVE` 판정을 내리고 거래를 즉시 승인합니다. 실제 테스트에서 응답 시간이 약 15~30ms 수준으로 측정되었음.

### 4-4. ML 모델이 없을 때의 폴백 전략

모델 파일(.pkl)이 준비되지 않은 상황에서도 서비스가 중단되지 않도록 **Rule-Based 폴백 엔진**을 함께 구현하였음. 이 엔진은 잔액 초과 여부, 거래 금액 수준, 속도 체크 등의 규칙을 조합하여 위험 점수를 산출합니다. 실제로 ML이 배포되지 않은 환경에서도 1차 방어선 역할을 할 수 있어, 시스템의 안정성을 높이는 데 기여하였습니다.

## 5. 기대 효과 및 느낀 점

### 5-1. 기대 효과

**보안 강화 측면**

ML 기반 탐지는 기존 규칙 기반 FDS로는 잡을 수 없었던 복잡한 사기 패턴까지 학습하고 대응할 수 있음. 특히 99.76%의 사기 탐지율은 실제 운영 시 사기꾼이 시스템을 우회하기 매우 어렵다는 것을 의미합니다.

**피해 최소화 측면**

CRITICAL 등급 거래의 경우 계좌 동결까지 자동으로 수행하기 때문에, 사람이 개입하기 전에 이미 피해 확산을 차단할 수 있음. 특히 새벽이나 휴일처럼 담당자가 즉각 대응하기 어려운 시간대에 그 효과가 더욱 두드러집니다.

**신뢰성 확보 측면**

판단 근거를 자연어로 함께 반환하는 XAI 기능 덕분에, 오탐이 발생하더라도 담당자가 근거를 바탕으로 빠르게 검토할 수 있음. AI의 결정을 무조건 따르는 것이 아니라, 사람과 AI가 협력하여 최종 판단을 내리는 구조가 **신뢰성 확보**의 핵심이라고 생각하였습니다.

**사회적 기여 측면**

금융 범죄 예방을 통해 안전한 디지털 거래 환경을 조성하고, 나아가 비대면 금융 서비스에 대한 사용자의 신뢰도를 높이는 데 기여할 수 있을 것으로 기대합니다.

### 5-2. 개발 과정에서 어려웠던 점과 배운 것

**어려웠던 점 1 — 데이터 불균형**

처음에는 SMOTE만 쓰면 될 것이라 생각했는데, 630만 건에 바로 SMOTE를 적용하니 메모리 부족으로 Colab 런타임이 자꾸 죽었음. 언더샘플링을 먼저 적용한 후 SMOTE를 적용하는 하이브리드 전략을 찾아내기까지 꽤 많은 시도가 필요했습니다. 이를 통해 단순히 라이브러리를 사용하는 것보다 **왜 그 방법을 쓰는지** 이해하는 것이 훨씬 중요하다는 것을 배웠음.

**어려웠던 점 2 — 평가 지표 선택**

모델 성능을 어떤 지표로 평가할 것인지를 결정하는 것이 생각보다 어려웠음. 처음에는 Accuracy만 봤다가, 이 지표가 불균형 데이터에서 얼마나 큰 함정인지를 직접 경험하였습니다. Recall, Precision, F1-Score, ROC-AUC, PR-AUC 각각의 의미와 상황에 맞는 사용법을 깊이 이해하게 된 계기가 되었음.

**어려웠던 점 3 — 백엔드 설계**

FastAPI를 처음 써보면서 Pydantic 스키마 설계에 많은 시간을 썼음. 특히 실제 은행 전문 구조(Header/Body/Metadata)를 JSON으로 표현하는 방식이 낯설었는데, 이 과정에서 금융 시스템의 데이터 전문 규격에 대해 이해도가 많이 높아졌습니다. 외래키(Foreign Key) 제약 조건으로 인한 DB 오류를 디버깅하면서 데이터 무결성의 중요성도 체감하였음.

## 6. 향후 계획

### 6-1. XAI(설명 가능한 AI) 강화

현재 구현된 XAI는 규칙 기반으로 판단 근거를 텍스트로 생성하는 수준임. 이를 SHAP(SHapley Additive exPlanations) 라이브러리를 활용한 피처 기여도 분석으로 업그레이드할 계획입니다. SHAP을 사용하면 "이 거래에서 금액이 최종 점수에 얼마나 기여했는지", "신규 디바이스 여부가 얼마나 영향을 미쳤는지"를 수치로 표현할 수 있어, 진정한 의미의 **설명 가능한 AI**를 구현할 수 있음.

### 6-2. Streamlit 대시보드 연동

현재 FastAPI 백엔드만 구현된 상태임. `GET /api/v1/fds/stats` 엔드포인트를 활용하여 Streamlit 기반의 실시간 모니터링 대시보드를 구축할 계획입니다. 위험 등급별 거래 분포, 시간대별 탐지 추이, 평균 처리 시간 등을 시각화하면 시스템의 완성도를 높일 수 있을 것으로 기대함.

### 6-3. FK 버그 수정 및 DB 고도화

현재 Swagger UI 테스트 시 DB에 없는 계좌번호로 요청하면 Foreign Key 제약 오류가 발생하는 문제가 있음. `INSERT OR IGNORE` 방식으로 수정하여 미등록 계좌의 거래도 안정적으로 처리할 수 있도록 개선할 예정입니다.

### 6-4. 모델 성능 고도화

현재 모델은 정적인 거래 데이터 기반이지만, 실제 금융 환경에서는 고객의 거래 시퀀스(순서)도 중요한 단서가 됨. 향후에는 LSTM 등 시계열 모델을 접목하거나, 그래프 신경망(GNN)을 활용하여 계좌 간 네트워크 분석을 추가하는 방향도 검토하고 있습니다.

### 6-5. 실제 배포 환경 전환

현재는 SQLite와 로컬 서버 환경에서 동작하는 프로토타입 수준임. PostgreSQL 연동, Docker 컨테이너화, 그리고 클라우드 배포까지 완성하면 실제 서비스에 가까운 구조가 될 것입니다. 다음 학기 심화 프로젝트나 개인 포트폴리오로 이어나갈 계획이 있음.
